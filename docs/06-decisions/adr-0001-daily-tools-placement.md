# ADR-0001: Tools 区服务是否合并部署到单一 LXC/VM

- Status: Accepted
- Date: 2026-02-22

---

## Context

当前 homelab 架构：

- 单节点 Tiny P3（PVE）
- 单网段（无 VLAN）
- tools 区：192.168.5.20–39
- 服务类型包括：
  - 反向代理（nginx / caddy / traefik）
  - 监控（Prometheus / Grafana）
  - DNS（AdGuard / Pi-hole）
  - 日志（Loki / ELK）

问题：

> 这些“日常基础工具类服务”，是放在同一个 LXC/VM 中，还是拆分到多个 LXC/VM？

需要在以下维度之间权衡：

- 隔离性
- 故障域大小
- 资源利用率
- 运维复杂度
- 回滚难度
- 学习成本

---

## Decision

**每个核心基础服务使用独立 LXC 容器部署。**

具体建议：

- 192.168.5.20 → 反向代理
- 192.168.5.30 → DNS
- 192.168.5.31 → 监控
- 192.168.5.32 → 日志

不将多个功能混合在同一个 LXC/VM 中。

优先使用 LXC（非完整 VM），除非有内核级隔离或特殊需求。

---

## Consequences

### 收益

1. 故障域隔离  
   - 监控挂掉不会影响反向代理
   - DNS 崩溃不会拖垮日志系统

2. 升级风险最小化  
   - 升级某个服务不会影响其他服务
   - 可独立快照 / 回滚

3. 安全边界更清晰  
   - 不同服务最小权限运行
   - 减少横向影响

4. 可观测性更强  
   - 资源占用可单独统计
   - 排障路径清晰

5. 更符合未来扩展  
   - 未来增加 VLAN 时可直接映射为独立安全域

---

### 代价

1. 管理对象增多  
   - 多个 LXC
   - 多个 IP

2. 初期部署稍复杂

3. 资源占用略高（但 LXC 开销极低）

---

### 风险

- DNS 如果和监控分离，可能导致“DNS 故障影响所有服务访问”
- 单节点环境下，宿主机仍是单点故障（不可避免）

---

## Alternatives considered

### 方案 A：全部合并到一个 LXC（未选择）

优点：

- 管理简单
- IP 少
- 部署快

缺点：

- 故障域过大
- 升级风险扩大
- 安全边界模糊
- 排障复杂

结论：

> 适合极小规模演示环境，不适合长期 homelab。

---

### 方案 B：全部使用 Docker Compose 放在一个 LXC 内（未选择）

优点：

- 服务逻辑隔离
- 管理方便

缺点：

- 容器层叠（PVE LXC + Docker）
- 网络复杂度上升
- 排障时多一层抽象

结论：

> 可行，但不如“一服务一 LXC”边界清晰。

---

### 方案 C：全部使用独立 VM（未选择）

优点：

- 最强隔离

缺点：

- 资源浪费
- 管理复杂
- 对 Tiny P3 性能不必要消耗

结论：

> 对 homelab 属于过度工程化。

---

## Rollback plan

如果“一服务一 LXC”带来管理负担过高：

1. 将低风险服务（如日志 + 监控）合并到同一个 LXC
2. 保持 DNS 和反向代理独立
3. 更新 IP 映射与文档
4. 删除多余 LXC

回滚成本低，影响范围可控。

---

## 最终原则总结

- 核心基础设施 → 独立 LXC
- 高风险服务 → 独立 LXC
- 临时实验 → 可合并
- 不使用单一“大杂烩容器”

> 在单节点 homelab 中，“小而清晰的故障域”比“极简数量”更重要。
